03会中

运行流程

当前主路径是手动邀请 Bot。agent 消费、会中弹幕和语音都挂在同一场会议 session 上，结束后统一收尾。

1

用户邀请 bot 入会

meeting_invited 到达 daemon，视为显式授权，不再二次私聊确认。

2

bot 用指定 profile join

join 必须绑定当前 listener bot 的 larkCliProfile，禁止落到 lark-cli 默认 profile。

3

创建监听群

监听群用于同步稳定字幕、会中聊天、参会变化、输出审批卡和 agent 讨论。

4

activity ingest

push 和 polling fallback 写入同一份 per-meeting state，按 sentence_id/message_id 去重。

5

ended 收尾

最终 flush、发送结束记录、清 runtime record，并用 tombstone 拦住迟到 activity 复活 session。

卡片选择

监听群发 agent 选择卡。下拉只暂存，点击“确认”才生效；可选择只监听，也可选 30/60/120 秒同步间隔。

跨 daemon 注入

若目标 agent 不在会议 bot daemon 内，会议 daemon 调目标 daemon 的 /api/trigger，保证注入和用户 @agent 落入同一个 chat-scope session。

单 tick 同拍

同一个 timer 周期先 flush 监听群，再评估 agent 注入，避免群消息和 agent 上下文存在固定相位差。

catch-up

用户在监听群 @agent 时，会先 best-effort 拉一次最新稳定会议内容，再放行用户 turn。

会中弹幕不是监听群消息

agent 通过 botmux vc-agent request-output --channel text 提交请求，daemon 发审批卡。授权人批准后走 lark-cli vc +meeting-message-send --msg-type text，消息出现在会议聊天/弹幕里。

  • 默认需要审批；本场可切到自动文本。
  • 失败时卡片展示真实错误；ok:false 不会被误判成功。
  • 发送 uuid 使用 requestId，避免重复发送。

实时语音由 daemon 统一执行

agent 只能提交语音请求，不能直接连实时音频 WS。授权人批准后，daemon 复用会议 session 持有的 voice session，合成 PCM 后按 100ms 帧推给会议。

  • 需要 vc:meeting.bot.realtime:write 和会议侧“允许 AI 智能体发言”。
  • 实时协议使用 session.create 握手，等 session.created 后才推音频。
  • 失败可降级成会中弹幕或监听群文本，取决于审批选择。

runtime store

已开始监听的会议会保存 meetingId、listenerChatId、consumer 选择、输出策略和同步间隔。

恢复策略

daemon 重启后恢复监听路由和 timers；不恢复待审批 nonce。全局开关关闭时跳过恢复。

重启边界

已 ACK 未 flush 的最近窗口内容可能丢失；后续可通过 restore-time polling reconcile 补偿。

僵尸保护

ended 事件和 close 都会移除 runtime record；迟到 activity 用 ended tombstone 拦截。

05输出门

安全与审批

会议内容属于不可信输入。防护重点不是让 agent “更听话”，而是把会中对外动作收敛到 daemon 审批门。

默认输出策略

  • 文本：默认审批，可在本场切到自动文本。
  • 语音：默认禁止或审批；不建议在 dogfood 期开放免审。
  • 权限只是意愿，实际能力还受 scope、会议侧开关和 endpoint 可用性硬门约束。

只认授权用户的指令

  • 会议转写按 speaker 标注“授权用户/仅上下文”。
  • 非授权用户说出的命令只作为上下文，不作为 agent 指令执行。
  • 声纹归因不是密码学保证，所以审批卡仍是外发动作的硬门。

审批卡

  • 同意语音、改发弹幕、拒绝。
  • 单 channel 同时只保留一个 pending，请求超时自动拒绝。
  • 审批结果会回注给 agent，避免重复请求。

隐私边界

  • raw transcript/chat 默认不长期落盘。
  • 监听群是会议内容可见面，群成员必须明确。
  • 主动 ambient 监听暂缓，避免每用户 OAuth 和隐私撤销复杂度。
边界
daemon 门能挡住“会场话术注入 agent 后经正常工具面发声/发弹幕”的主威胁；它不是进程级沙箱。若 agent 进程被视为完全不可信，需要另做沙箱和密钥隔离。

07答疑

常见问题

为什么不默认监听用户加入的所有会议？

这需要用户态事件/OAuth、token 刷新、撤销、隐私告知和用户到 bot 的路由。相比“手动邀请 bot”只多了 ambient 便利性，MVP 阶段性价比不高。

监听群里的 agent 为什么能回答“刚才谁提到 X”？

会议内容注入和用户 @agent 都落到目标 agent daemon 的同一个 chat-scope session。用户 @ 时还会先做一次 best-effort catch-up，减少“我听到了但 agent 还没 flush 到”的时序差。

会中弹幕和监听群文本有什么区别？

会中弹幕通过 VC meeting message API 发到会议聊天里，所有会议参会人可见；监听群文本只在 botmux 创建的监听群里可见。

为什么输出需要审批？

会议转写可能包含提示词注入，speaker 归因也不是强认证。默认审批把“对真实会议说话/发消息”的执行权放在 daemon 和授权用户手里。
